14081: ef9345: scanline generation should not assert/deassert the BUSY flag [fabio-d]
The status register's BUSY flag should only be asserted by EF9345 while a user command is pending or being executed. Rendering (i.e.
  generation of the next line of pixels) is an independent process and it should neither set it nor clear it:
<img width="45%" alt="BUSY is set at the beginning of any command execution. It is reset at completion"
  src="https://github.com/user-attachments/assets/5fc21d22-5f06-46d6-879c-90de9c69097c" />
The current code, instead, always asserts it at the beginning of the scanline generation periodic task and deasserts it shortly
  afterwards (after a fixed timeout). This causes the real BUSY value and timer, that were previosly configured as a result of user
  commands in `ef9345_exec` calling `set_busy_flag`, to be lost.
Figure 10 of the EF9345 datasheet shows that, in real HW, rendering just delays memory accesses due to user commands by 104 us
  (=40+24+40). As an approximation, this commit delays the completion of the current command, irrespective of whether it involves
  memory accesses or not.
<img width="45%" src="https://github.com/user-attachments/assets/e80db67d-dec3-4900-b546-908124ee00b5" />

14167: Fix regression in apple2gsr1 with large ramsize [arekkusu42]
This PR fixes a regression introduced in MAME 0.279, which causes OOB corruption in both the emulated ROM1 Apple IIgs and host
  machines.
To reproduce:
1) launch MAME with: `apple2gsr1 -ramsize 4M -sl7 cffa2 -hard1 gsos.2mg` (_6.0.4 via http://www.whatisthe2gs.apple2.org.za/ or
  similar_)
2) OA-Ctrl-Esc to access classic control panel, or
3) quit MAME
The bug is caused by OOB r/w, so the results will be host/OS-specific, but on my arm64 Mac running Sonoma 14.7.8, it reproducibly
  manifests as text corruption in the classic control panel, and when quitting MAME, as a "pointer being freed was not allocated"
  crash (within `audio_effect_reverb::~audio_effect_reverb()`):
<img width="560" height="384" alt="ControlPanelCorrupt"
  src="https://github.com/user-attachments/assets/ea093788-7e95-4705-8a37-8c8bdba4cbbf" />
Recompiling with `-fsanitize=address`, ASan immediately hits heap-buffer-overflow, writing past the end of the IIgs's `m_ram`
  allocation:
mame_280_asan.txt
The bug here was exposed by bfb18d1 using `m_ram_size` to bound r/w.  But the cause was in 2fb9b7c: notice that for `(!m_is_rom3 &&
  m_ram_size > 1280K)`, there is no adjustment made to `ramsize`, resulting in `install_ram()` being called with `addrend` 128k past
  the original size.  Also, the adjustments are made to a local variable, not `m_ram_size`.
The fix committed here simplifies that logic to only adjust `m_ram_size`. This results in the same total RAM sizes on ROM1/3 as
  prior to bfb18d1, and fixes the OOB. The 8 MB adjustment is no longer required as that is handled in the recently-added
  `expandedram_r/w()`.  The actual allocation within `m_ram` is *not* adjusted, it can still waste 128k the same as before.
Testing: MegaMemoryTester helpfully prints the total (slow+fast) and detected expansion card RAM and corresponding start/end banks.
Noting the remaining (preexisting) oddities here:
* a "1M" ROM0/1 gets 128k slow + 128k fast + 768k expansion = exactly 1M, while a "1M" ROM3 gets 128k slow + 1M fast = 1.125M.
* \> 1M on a ROM3 results in realistic expansion card sizes (128k slow + 1M fast + 4M expansion, for example), but unrealistic sizes
  on a ROM0/1 (128k slow + 128k fast + 4992K expansion.)  Fixing this would require *increasing* the `m_ram` allocation by 128k; I
  don't see a simple way to do that (i.e. in a `ram_device` subclass) while `m_pointer` is private.
* ultimately, the -ramsize options are a bit confusing because the >= 2M options work slightly differently than < 2M.  But even
  Apple's own documentation is self-contradictory in how it describes total RAM, so perhaps MAME is in good company?

-Basic fix for pc_chd support in imgtool (#14148) [Michael Karcher]
 * imgtool: Take ownership of stream in pc_chd_image_open
   - `imgtool::image::internal_open` passes an rvalue reference to the stream to the `open` function of the image format module. It
      expects the `open` function to take ownership if it keeps a reference to the stream. If `open` does not do so, the `stream` is
      going to be destroyed at the end of `internal_open`.
   - `pc_chd_image_open` fails to take ownership, yet it persists a reference to the stream as part of `info->hard_disk`. This
      causes an use-after-free condition
 * imgtool: Correct determination of total sectors of a FAT volume
   - The number of total sectors of a FAT volume is stored either in the 16-bit word at offset 19 or, if that word is zero, in the
      32-bit word at offset 32 instead. The 32-bit word is not a high word to build a 48-bit value in conjunction with the 16-bit
      word at offset 19, but it supersedes it.
