9546: Fixed some issues involving fs::meta_description construction [npwoods]
With the recent change to use std::variant more closely, I noticed a problem where meta_descriptions of type meta_type::String got
  defaults of type 't'.  This was because the templated ctor for meta_description would convert 'const char *' to 'bool'.  This
  change adds another overload to catch 'const char *', along with asserts to catch problems.
In the process I corrected a few meta_description ctors
It is possible that this change does not go far enough.  Perhaps the meta_type argument should be removed, and we should instead
  create distinct ctor types (rather than relying on templates) and specify the precise meta_type in the overload.  Or even go
  further and remove m_type from meta_description, and instead create an overload that calculates meta_type based on calling
  std::visit on the variant

9554: k573dio/k573fpga: Improve counters and status flags [987123879113]
MAS3507D previously used a callback to receive data when it needed to fill the buffers which worked for the previous implementation
  but wasn't accurate to what the MAS3507D actually does, which is trigger the demand pin and wait for the other end to send data
  via the input clock and input data pins. I rewrote that code to more accurately reflect the real MAS3507D which allowed me to
  implement the FPGA side more accurately to what the real code was doing based on my research with the state flags and
  timers/counters.
Datasheet ref: http://www.mas-player.de/mp3/download/mas3507d.pdf
This PR also closes https://github.com/mamedev/mame/issues/8619. Further research showed that DDR Solo Bass Mix times songs entirely
  differently than any other digital I/O game. I had previously identified that the register used for mpeg_key_1_w is also read at
  times so I added in a mpeg_key_1_r, but upon further research on real hardware this turned out to be the MP3 frame counter which
  is incremented when the MAS3507D decodes a frame and triggers the MPEG-FRAME-SYNC pin (PI4). DDR Solo Bass Mix syncs its internal
  game timer to the first 1 second worth of frame syncs and then uses the regular timer counter to increment the game's internal
  timer after that. Additionally, the timer itself has been confirmed on real hardware to be an always running timer, in comparison
  to the normal FPGA code which starts, stops, and resets the timer at specific events.
Audio timing using the internal MAME video recorder looks to be near spot on with these changes. Externally using probably the best
  case scenario for audio (PortAudio + WASAPI) the timing is improved but there is still a slight amount of latency when actually
  playing. If the FPGA's counter is offset by ~28ms to compensate for latency then it improves the timing to be close enough to the
  real arcade machine to be a non-issue based on testing from a few players.

-Revert initialisation of device members in headers. [Vas Crabb]
 * This is problematic in several ways:
   - Initialising things at construction that aren't needed until after start slows down -romident, -validate, -listxml, etc.  Slot
      cards can be a particular drain on -listxml and -validate as they're instantiated for every compatible slot.  It's more
      pronounced for array members, too.
   - Splitting member initialisation between declaration in headers and constructors in source files means you have to look at two
      places to check for the initial value, and you always need to check the constructor even if an initialiser is present in the
      header because the constructor initaliser list takes precedence.  (This isn't as much of an issue for driver classes because
      the constructor is most often inlined at declaration, so it isn't far from the member declarations.)
   - Initialisers in headers for frequently-used devices increases the frequency of recompiling dependent devices/drivers as they're
      exposed to any changes in initialisers.
   - Initialisers in frequently-used headers increase build times because there's more for the compiler to parse/cache.  (This
      affects makedep.py as well for single-driver builds, but that's a single pass.)  It's not a lot individually, but it adds up
      given the size of MAME, which keeps increasing.  We've already had one contributor banned from GitHub actions for resource
      usage, we don't want to waste compiler time unnecessarily.
