7721: firebeat: Large refactor + initial beatmania III support [987123879113]
This is a large refactor of the Firebeat driver as discussed in #7693 so please forgive me for the huge PR with a meaningless diff.
  I've tried to make note of anything below that I've changed during the refactoring that wasn't just structural refactoring.
All games on the Firebeat platform except those with missing HDD data (`bm36th` and `bm3core`) have been tested to make sure they
  are still booting into gameplay and inputs responding as expected.
All of the required new ROMs for this PR have been sent to `code ( at ) mamedev`.
NOTE: Even with this PR, pop'n music *will not* boot as-is currently due to lack of support for the DVD CHDs. The CHDs are made as
  HDD CHDs which can't be loaded by the CD-ROM emulation code. The pop'n music games will work if you extract the raw data from the
  existing CHDs and rebuild them using `chdman createcd` but that is just a workaround. I will try to fix this in a future PR.
NOTE 2: RTC initialization for pop'n music, `bm3`, and `bm3core` are locked behind a randomly generated password for which you were
  supposed to call Konami to get a password that corresponds to your dongle serial ID + seed. This is not a bug, it's a feature
  (thanks Konami). You will need to use a tool I made to generate the required passwords.
Source code: https://github.com/987123879113/firebeatrtc
Web version: https://987123879113.github.io/firebeatrtc/
Ideally there should be a way to include a clean initialized RTC file in the ROM dumps so as to not require everyone to input a
  password but I couldn't find a way to do that with the RTC65271 so that's on hold for now.

Changelog:
- firebeat: Refactored entire monolith `firebeat_state` into (hopefully) logical components based on machine configurations. The
  Firebeat system is highly modular and each individual game has its own machine configuration, sometimes with the same boards being
  populated differently between games. I attempted to recreate that in the code so that only the components that are required for
  each game are loaded.
Also discussed in #7693, in the future I will look at splitting the SPU code off into its own machine to be shared with Twinkle
  because the two have nearly identical SPUs but that will be for another PR.
- firebeat: Rewrote `cabinet_r` register
Every game treats this a little differently so the initial value needs to be set based on cabinet type, which includes whether the
  machine is an overseas or Japanese machine, a rental or a purchase machine, and some other bits to determine actual machine type.
The values picked have all been checked by debugging the games to verify happy paths without requiring any additional hacks. These
  values are also tied to the specific dongle used in some cases which is why "normal" type dongles have been provided to replace
  all of the old non-"normal" type dongles (see below for more details).
- firebeat SPU: Changed dipswitch register to use dipswitch inputs instead of a function because the dipswitches are accessible on
  the board
- firebeat SPU: Added documentation for unknown parts of SPU code (IRQs 1 and 2, register 0x220000)
- firebeat SPU: Added attempt to ease blocking issues between audio CPU and main CPU during DMAs.
DMA handling is done entirely by the audio CPU. It's expected that the audio CPU can perform DMAs asynchronously apart from the main
  CPU in order to refill the BGM sound buffer while songs are being played.
The existing DMA code from my previous PR works as a bare minimum implementation but will block during large DMAs (size >=
  0x1000000) which are frequent in beatmania III and pop'n music.
- firebeat ParaParaParadise/beatmania III: `sensor_r` has been changed from a 32-bit register to 16-bit based on the way beatmania
  III reads the turntable's values as two separate 16-bit values (upper and lower). The appropriate changes were made to PPP's
  sensor inputs and tested to make sure inputs still worked.
- firebeat Keyboardmania: Swapped keyboard inputs to match I/O test menu (P1 should be left screen, P2 should be right screen now)
- firebeat beatmania III: Added new game: `bm3` added as `MACHINE_NOT_WORKING`
- firebeat beatmania III: Added proper SPUs/ROMs for all beatmania III games
- firebeat beatmania III: Added basic I/O emulation. Does not support Zoom ST-224 effects board or FDD.
- firebeat beatmania III: A hack (similar to what was used until recently in System 573 for `ddr2ml`/`ddr2mla` security plate
  errors) was added to bypass the FDD errors in all beatmania III games. The FDD only adds saving and loading scores and will
  otherwise run without errors if it's patched out. The patches have been tested on all beatmania III games. The patches will be
  removed when FDD emulation is implemented enough that the games can be loaded without the patches.
- firebeat pop'n music: Added new games: `popnmt`, `popnmt2`, `popnanm` added as `MACHINE_NOT_WORKING`
- firebeat ROMs: Replaced/added dongles for the following games
    - `ppp`
    - `ppd`
    - `ppp1mp`
    - `ppp11`
    - `kbm`
    - `kbm2nd`
    - `popn6`
    - `popn7`
    - `popn8`
    - `popnanm2`
    - `bm36th`
    - `bm37th`
    - `bm3final`
    All of the replacement dongles are generated through a tool, not dumped from security keys. All generated dongles have also been
  marked as BAD_DUMP to be sure that they don't get mixed up with real dongles. All dongles have been tested to make sure they boot
  in-game as expected.
    A large majority of the previous dongles dongles were also manually generated instead of dumped from the real security keys. The
  reason the old ones needed to be replaced is because the dongle type used by the old dongles was often the wrong one (service or
  manufacture type instead of a normal dongle) which in some cases caused the game to not load properly or in a way that it would
  not normally on real hardware with real dongles. There are legit dongle dumps in MAME already (`kbm3rd`, `popn4`, `popn5`,
  `bm3core`) which were used as references to generate as accurate as possible dongles. The main differences being the serial number
  and the lack of 8 bytes at the end of each file, which are not used at all during emulation, also data which the previous fake
  dongles lacked. An easy way to tell apart the fake from real dongle dumps is the fakes are 0xc0 bytes in size while real are 0xc8
  bytes in size.
Other:
- k057714: Adjusted transparency blend mode based on new info from beatmania III
- k057714: Fixed some crashes (divide by zero, out of bounds)
- k057714: Clear vram between resets

7741: Reinstated RI Handler [fergusleen]
The COMMSTAR ROM software won't run due to the ring indicator bit 4 of the DART RR0 being high when it's expected to be 0. 
If line 55 is uncommented bit 4 is low as expected and the software runs.
I've tried a few different pieces of mid 80's comms software and all is good.
Reading between the lines of forum posts back in 2014. It seems that the z80 dart emulation has been improved since then and I think
  now's the time to reinstate that line.
The routine in cpc_ser/COMMSTAR1.ROM that reads the z80 dart RR0 are:
```
               LD      BC,&FADD                ;;F102:   ...     01 DD FA
                LD      A,&18                   ;;F105:   >.      3E 18
                OUT     (C),A                   ;;F107:   .y      ED 79
                LD      A,&10                   ;;F109:   >.      3E 10
                OUT     (C),A                   ;;F10B:   .y      ED 79
                LD      A,&00                   ;;F10D:   >.      3E 00
                OUT     (C),A                   ;;F10F:   .y      ED 79
                IN      E,(C)                   ;;F111:   .X      ED 58.  --- READ REGISTER 0 Expecting 6C getting 7C
                LD      A,&56                   ;;F113:   >V      3E 56
                AND     E                       ;;F115:   .       A3
                CP      &44                     ;;F116:   .D      FE 44 -- replace with &56 to work with &7C
                LD      A,E                     ;;F118:   {       7B
                SCF                             ;;F119:   7       37
                CCF                             ;;F11A:   ?       3F
                RET     NZ                      ;;F11B:   .       C0 
```

7766: bgfx crt-geom and crt-geom-deluxe enhancements [cgwg]
Contains three changes to these shaders:
1. A "brightness boost" feature for the shadow mask that works by making the brightness ratio between bright and dark mask pixels
  closer to 1 for the brighter parts of the image. I've been careful to ensure that the average of bright and dark mask pixels still
  varies linearly, so that the tone curve is unaffected as long as the gamma of the output display is set correctly.
2. Clamping to zero so that underscanning produces a black border.
3. In crt-geom-deluxe, a "raster bloom" effect that makes the image grow slightly when the average brightness of the screen is high.
  This mimics a common defect in CRTs and has been previously implemented in the shader called crt-guest-dr-venom. I find the effect
  kind of nice when the image fades in from black.
One detail I am uncertain about comes from the brightness boost feature: this required encoding some extra information in the alpha
  channel of the shadow mask textures. I wrote a very simple Python script to do this and have included this script, but perhaps
  this is inappropriate or it should be put in a different directory.
Finally: the shaders have only been compiled for glsl; someone else will have to generate the full set of shader targets.
