15003: tmp94c241: Implement DMA subsystem (HDMA + DMAR) and fix port reads [felipesanches]
Implement the TMP94C241's DMA subsystem:
- **HDMA:** Hardware DMA triggered by interrupt events. Transfers one unit per trigger, with priority over normal interrupt
  dispatch. Channels checked in priority order (0 highest). Supports all DMAM transfer modes (byte/word/long,
  increment/decrement/fixed) using switch-based decoding matching the proven TMP95C061 implementation.
- **DMAR:** Software DMA trigger register at SFR 0x109. Writing bit N triggers one DMA transfer on channel N. For example, the
  Technics KN5000 driver writes DMAR once per INT0 to receive bytes from the inter-CPU latch one at a time.
- **HDMA priority in check_irqs():** Interrupts targeted by active HDMA channels are consumed by the DMA engine instead of
  dispatching to the interrupt handler.
- **INT0 level-detect re-assertion:** After clearing an INT0 flag during IRQ dispatch, re-assert it if the input pin is still active
  and level-detect mode is configured. This matches real hardware behavior where level-triggered flags are continuously driven.
- **Port read fix:** `port_r()` now returns the output latch value for bits configured as output and the external pin level for
  input bits. Previously it always returned the external pin level, which broke firmware that reads back its own output state.
- Add debugger `interrupt_hook()` notification on IRQ dispatch.

15015: tmp94c241: Add serial port sub-device with I/O interface mode [felipesanches]
Replace the inline serial stubs (which just instantly set TX-complete)
with a proper sub-device implementation supporting:
- **I/O interface mode (mode 0):** Synchronous clocked serial using SCLK
  pin. Supports both internal (baud rate generator) and external
  (IOC=1) clock sources. TX double buffering with auto-load from
  buffer to shift register. Pre-outputs bit 0 on TXD before first
  clock edge so receiver can sample it on the rising edge.
- **Baud rate generator:** Configurable via BRxCR register with divisor
  and clock source selection. Timer drives SCLK at the configured
  rate. Clock frequency derived from CPU clock.
- **TX/RX data callbacks:** txd(), rxd(), sclk_out(), sclk_in() for
  connecting external devices to the serial ports.
- **tx_start callback:** Signals the start of each byte transmission with
  the current PFFC pin function state, allowing connected devices to
  distinguish real transmissions from phantom ones.
The serial registers (SC0BUF/SC1BUF, SC0CR/SC1CR, SC0MOD/SC1MOD,
BR0CR/BR1CR) are now delegated to the sub-devices in the internal
memory map. TX-complete flags (INTES0/INTES1 bit 7) are set at
device_reset to indicate empty TX buffers at power-on.
UART modes (7/8/9-bit) are recognized but not yet implemented.
Also moves interrupt register indices from a file-scope enum to
public static constexpr members of tmp94c241_device.

15064: machine/pit8253.cpp: Improved timing accuracy. [m1macrophage]
Based on @rb6502's suggestion here.
Matched the mode descriptions and timing diagrams in the 8254 datasheet. The "Operation Common to All Modes" section summarizes most
  behaviors applicable to this PR.
This change **only** affects configurations where CLK is driven by **`write_clk*()`**. There should be **no change** in behavior for
  configurations that use **`set_clk()`**.
Specifically:
1. The counter is loaded and decremented on the falling edge of CLK.
1. GATE and GATE transitions (for edge-sensitive modes) are sampled on the rising edge of CLK.
   * Ensures GATE changes affect the counter on the next full CLK cycle (rising + falling edge).
1. Mode 3 treats GATE as edge-sensitive.
1. In modes 2 and 3, GATE has an immediate effect on OUT.
1. Deferred the execution of `set_clock_signal`.
   * Matches behavior of other control functions, and ensures calls to `set_clock_signal` are sequenced correctly.
The tuning algorithm on the `sixtrak` depends on (1) and (2) to function correctly, so it had local workarounds. Those have been
  removed.
-------
**Risk analysis and mitigation**
Given the 8253 is used in many places, I consider this a risky change. The blast radius is somewhat reduced due to `set_clk`
  configurations not being affected. I've also written **unit tests** replicating all waveform examples in the 8254 and 8253
  datasheets, along with the tuning sequences of the `sixtrak`. Those can be found here. If interested, I can send a PR for those.
  Manual tests were done on a handful of synths and the sente6vb.
A list of potentially affected devices and drivers can be obtained with:
```
grep -r 'write_clk[012]' src/ | awk -F: '{print $1}' | sort -u
```
Narrowing down to just driver files, ~10 out of ~25 have at least one working system. The others only have non-working systems.
There are some remaining, preexisting issues:
* If GATE is low when the counter is written, the counter won't get initialized until GATE goes high, even though it should be. This
  affects a couple of modes. See Figure 19, "Mode 4", second example, in the 8254 datasheet.
* Fix (4) above was intentionally left out of `set_clk` configurations.
* A counter write will initialize the counter earlier than it should.
Fixing these would affect the `set_clk` configuration and/or require more invasive changes, so I left them out of this PR. I expect
  they would only make a difference in contrived cases. I can revisit if/once this PR has been live for a bit.

Code comments:<details>
In fixing the HBL timing, I have chosen to leave `set_raw()` with HBL to the right like it was previously, and adjust all uses of
  `hpos`/`vpos`/`vblank`.  The alternative is to properly place HBL to the left in `set_raw()`, but MAME's `screen()`
  unconditionally allocates the entire bitmap width including HBL.  For the apple2 machines this is wasted space, which then
  requires re-adding a (constant!) offset in each video update when manipulating each scanline (there's no method to adjust the base
  pointer of the bitmap allocation once on realloc?)  This gets worse when you also consider the apple2gs `m_8bit_graphics` which
  does not include HBL, so now we're continually re-adding and re-subtracting the same constant... ^_^;  I judged this
  implementation to be the lesser evil.  A wart is that the uncorrected `hpos`/`vpos` are still shown in MAME's debugger.
On apple2gs, screen updating has moved to the new `apple2_vgc` callback.  Happily, this occurs past the right border, so we still
  have 231 partial updates per frame like before.
On apple2gs (or apple2e/c with a Zip Chip), changing CPU clock speed introduces phase drift in the timer callbacks, which leads to
  off-by-one errors in the beam position.  To fix this, I am using the clock domain sync from #9726.  That works.  Sadly, it does
  not fix any of the disk corruption problems.

```cpp
// After:
m_irq_vector.fov = (m_vcrd >> 8) & 0x7f;
```
`vcrd_w` is the write handler for VCRD (0xFFFFFE68), which holds the FRT overflow interrupt vector in bits 14–8. The code was
  reading `m_vcrc` (the FRT input-capture / output-compare vector register) instead of `m_vcrd`. This meant writing VCRD never
  actually updated the FOV vector — it silently re-latched whatever was in VCRC. Any game or program relying on a custom FRT
  overflow vector via VCRD would get the wrong handler.
**2. `dmaor_w`: AE and NMIF flags could never be cleared**
```cpp
// Before:
m_dmaor = (data & ~6) | (old & m_dmaor & 6);
// After:
m_dmaor = (data & ~6) | (old & data & 6);
```
DMAOR bits 2 (AE — address error) and 1 (NMIF — NMI flag) are specified as "write 0 to clear" flags (§9.2.7). The original code
  computed `old & m_dmaor & 6`, but since `old` was assigned from `m_dmaor` on the previous line, this reduced to `m_dmaor & m_dmaor
  & 6` = `m_dmaor & 6`, unconditionally preserving both flags regardless of the written data. Once set, AE or NMIF could never be
  cleared, permanently disabling DMA (the DMAC requires both flags to be 0 to operate). The existing TODO comment on this line
  already suspected this was a bug. The fix uses `old & data & 6`, which is standard W0-clear behavior: writing 0 clears the flag,
  writing 1 leaves it unchanged.
**3. `vcrb_r`: reserved bits not masked on read**
```cpp
// Before:
return m_vcrb;
// After:
return m_vcrb & 0x7f7f;
```
VCRB (0xFFFFFE64) holds the SCI transmit and transmit-end interrupt vectors in bits 14–8 and 6–0 respectively. Bits 15 and 7 are
  reserved and should always read 0 per the hardware manual. Every other VCR register (`vcra_r`, `vcrc_r`, `vcrd_r`, `vcrwdt_r`)
  already applies the appropriate mask — VCRB was the only one missing it.

15121: m6502: route core fetch/data/write paths through interruptible accesses [dr-who]
This fixes a core integration gap in the 6502 family memory path.
While auditing the base NMOS 6502 against `perfect6502`, I built a small trace-comparison harness that checks bus activity
  event-by-event across representative instruction, addressing-mode, interrupt, and pause/stall cases. The unstalled traces matched,
  which ruled out a number of ordinary bus-sequencing errors in the core itself.
The actual problem turned out to be lower down in the memory integration layer: MAME’s wait/delay hooks are applied on
  `read_interruptible` / `write_interruptible`, but the base `m6502` core and several derived memory interfaces were still using
  non-interruptible fast paths for normal CPU-visible accesses. That meant memory-originated waits could be bypassed even when the
  core’s retry/defer machinery was otherwise correct.
A concrete example is `LDA abs` with a 2-cycle delay injected on the final data read:
- Correct timing:
   - final data read moves from cycle 3 to cycle 5
   - next opcode fetch moves from cycle 4 to cycle 6
- Before this patch:
   - the normal fast path could bypass the wait hook, so the read stayed at cycle 3 and the next fetch at cycle 4
- After this patch:
   - the access goes through the interruptible path, so the current bus cycle is stretched and the next fetch is correspondingly
  delayed to cycle 6
This patch:
  - routes base `m6502` opcode fetch, argument fetch, data read, data write, and refetch paths through the interruptible accessors
  - adds the needed opcode/data specific bindings so those paths use the interruptible dispatch consistently
  - updates the opcode generator so prefetch/refetch sites use the same wrapper path
  - audits the affected family variants with custom memory interfaces (`m6509`, `m6510`, `m4510`, `m50734`, `r65c19`, `rp2a03`,
  `st2204`, `st2205u`, `xavix`)
  - switches Apple II-family bus DMA hookups to `M6502_DMA_LINE` so DMA/bus denial is handled through the dedicated 6502 path rather
  than generic HALT suspension
Validation:
  - base/family interruptible source checks pass
  - `perfect6502` comparison suites still pass after the change
  - interrupt, pulse, pause, and stall semantics checks still pass
  - reduced and wider 6502-family compile coverage passed
  - patch is `git diff --check` clean
This is being submitted as a core correctness fix, not as a demonstrated end-user-visible game fix. I have not exhaustively tested
  games across the affected family, so I’m not claiming specific visible/audio improvements from this PR alone. The claim here is
  narrower: ordinary 6502-family bus cycles were not consistently reaching the interruptible memory path, so memory wait integration
  could be structurally invisible.

-tmp94c241: Add serial port sub-device with I/O interface mode (#15015) [Felipe Corrêa da Silva Sanches]
 * tmp94c241: Add serial port sub-device with I/O interface mode
   - Replace the inline serial stubs (which just instantly set TX-complete) with a proper sub-device implementation supporting:
     * I/O interface mode (mode 0): Synchronous clocked serial using SCLK pin. Supports both internal (baud rate generator) and
        external (IOC=1) clock sources. TX double buffering with auto-load from buffer to shift register. Pre-outputs bit 0 on TXD
        before first clock edge so receiver can sample it on the rising edge.
     * Baud rate generator: Configurable via BRxCR register with divisor and clock source selection. Timer drives SCLK at the
        configured rate. Clock frequency derived from CPU clock.
     * TX/RX data callbacks: txd(), rxd(), sclk_out(), sclk_in() for connecting external devices to the serial ports.
     * tx_start callback: Signals the start of each byte transmission with the current PFFC pin function state, allowing connected
        devices to distinguish real transmissions from phantom ones.
   - The serial registers (SC0BUF/SC1BUF, SC0CR/SC1CR, SC0MOD/SC1MOD, BR0CR/BR1CR) are now delegated to the sub-devices in the
      internal memory map. TX-complete flags (INTES0/INTES1 bit 7) are set at device_reset to indicate empty TX buffers at power-on.
   - UART modes (7/8/9-bit) are recognized but not yet implemented.
   - Also moves interrupt register indices from a file-scope enum to public static constexpr members of tmp94c241_device.
 * tmp94c241: move irq vector map to static const class member
   - Per galibert's review suggestion: defining the IRQ vector table as a static const member of tmp94c241_device and providing the
      out-of-class definition in class scope allows the INTE* constants to be referenced without tmp94c241_device:: qualifiers in
      the initializer.
   - The nested struct is named irq_vector_entry. NUM_MASKABLE_IRQS is replaced by std::size(irq_vector_map) directly at each use
      site inside member functions.
 * tmp94c241: use enum for public interrupt index constants
 * tmp94c241: use required_device for serial sub-device parent access
   - Replace dynamic_cast with required_device<tmp94c241_device> using DEVICE_SELF_OWNER to access the parent CPU from the serial
      sub-device. This provides automatic type checking during MAME's -valid pass.
   - Break the circular header dependency by replacing the #include of tmp94c241_serial.h in tmp94c241.h with a forward declaration,
      and having tmp94c241_serial.h include tmp94c241.h instead. The .cpp files include what they need directly.
