7548: intelfsh: Force fujitsu_29f016a to return status bits during clearing regardless of sector [987123879113]
This is a hack. The Fujitsu 29F016A datasheet specifically states "data polling must be performed at an address within any of the
  sectors being erased" whereas Konami is polling outside of the sectors.
All of the following information is based off pop'n music 6 in the Firebeat driver.
When initializing the flash memory, the following sequence of commands are issued in the following order:
1. Chip clear command issued, polling sector 0 until it's finished
2. Sector 0 clear command issued, polling sector 0 until it's finished
3. Sector 1 clear command issued, polling sector 0 until it's finished
4. Sector x clear command issued (looped until entire chip is erased again), polling sector 0 until it's finished
The polling code reads the first two bytes of the sector and XORs them together. The result is expected to be non-0.
Steps 1 and 2 work as expected. At step 3 the game errors out with a "FLASH RAM ERROR" message because it's erasing sector 1 but
  polling sector 0 which is returning `FF FF` instead of the expected status bits.

7664: Improve accuracy of System 573's digital I/O audio emulation [987123879113]
This PR is an attempt to improve accuracy of the System 573's digital I/O audio emulation. 
I noticed only 2 other devices in all of MAME even use STREAM_SYNCHRONOUS and I realize there's performance hit involved with its
  usage so I will offer a justification for its usage here. If there are other options besides STREAM_SYNCHRONOUS that would allow
  for accurately tracking playback at the sample level without the performance hit then I'd love suggestions.
The biggest issue currently with the k573dio/fpga emulation is audio buffering. The current sample counter implementation relies on
  the buffered sample counts: `u32 get_frame_count() const { return total_frame_count - buffered_frame_count; }`.
With this implementation, the sample counter increments in fixed intervals whenever the buffer is filled by sound_stream_update. I
  tried looking around in the sound-related interfaces and could only really find functions that returned the start and end
  positions of the current buffer but not the exact position being played within that buffer.
I ran various tests on real hardware to confirm that it's possible to read the counter at a per-sample level. I also checked that
  it's directly related to the number of samples and not something else like a standard timer just to be absolutely sure. On real
  hardware the sample counter increases proportional to the sample rate of the MP3. For example, 48 kHz will increase faster than
  44.1kHz, and a lower sample rate causes the sample counter to increase slower than the sample counter at 44.1 kHz.
One test I did that highlights the issue the clearest is a test that reads the sample counter register multiple times in succession
  with a vsync call in between. Using a MP3 at 44.1 kHz as an example here.
MAME results: The difference of sample counter updates in between vsyncs is always 882 with no variation.
Real hardware results: The difference of sample counter updates in between vsyncs is roughly around 736 with a variance of roughly
  15.
Konami uses the sample counter heavily for key parts of gameplay (animations and song timing), and the difference between what MAME
  returns and what the game expects results in noticeably inaccurate timings. As such, I turned the mas3507d device into a
  synchronous audio device so that every sample can be counted and an accurate sample count can be returned. This change brings the
  emulated code much closer to real hardware.
